## P20 (R15-Q4) — NAV staleness bound default

**Resolution:** **36h for LIVE, NULL for non-LIVE** (matches lean).

**Implementation locations:**
- Default value lives in `risk.upsert_limit` for drawdown-dimension limits at
  config time. Stored on `limit_versions.nav_staleness_bound_seconds` (column
  introduced in v1.4 R4 input lineage patch).
- LIVE: default `129600` (= 36 * 3600).
- non-LIVE: default `NULL` (no staleness bound enforced; backtest/replay
  tolerate any NAV age).

**Behavioral impact:**
- LIVE drawdown evaluation with `latest_nav.snapshot_at < as_of_at - 36h` →
  `severity_bucket='insufficient_inputs'`,
  `result_reason='insufficient_inputs:stale'` on the drawdown limit row.
- Per P15, this routes to `verdict_raw='degraded'` →
  `verdict_effective='blocked'` in LIVE.
- non-LIVE: NAV staleness check skipped entirely (NULL bound semantics).

**SQL-phase note:** The bound is configurable per-limit via the
`limit_versions.nav_staleness_bound_seconds` column. 36h is the default at
`upsert_limit` invocation; operators can override for specific limits.

## P21 (R15-Q5) — Throttle parameter schema

**Resolution:** **JSONB blob on `circuit_breaker_versions`** (matches lean).

**Implementation:**
- Existing column on `circuit_breaker_versions`:
  `throttle_params JSONB NULL` (introduced provisionally in v1.10's CB action
  taxonomy patch; now confirmed final).
- Required when `action = 'throttle'`; NULL otherwise.
- Schema interpretation owned by runtime, not the risk evaluator. Risk's job
  ends at "throttle CB is active for this evaluation, params are X"; the
  runtime decides what X means.
- Suggested params for runtime convention (not enforced by risk schema):
  `{"max_actions_per_window": int, "window_seconds": int}` or similar.
  Phase 1 does not lock a specific shape; runtime can evolve.

**Validation at config time:**
```
-- In risk.upsert_circuit_breaker, when action = 'throttle':
IF p_action = 'throttle' AND p_throttle_params IS NULL THEN
  RAISE EXCEPTION 'throttle CB requires throttle_params (JSONB)';
END IF;

IF p_action != 'throttle' AND p_throttle_params IS NOT NULL THEN
  RAISE EXCEPTION 'throttle_params only valid when action = throttle';
END IF;
```

CHECK constraint candidate for SQL phase:
```sql
ALTER TABLE risk.circuit_breaker_versions
  ADD CONSTRAINT chk_throttle_params_present_iff_throttle
  CHECK (
    (action = 'throttle' AND throttle_params IS NOT NULL)
    OR
    (action != 'throttle' AND throttle_params IS NULL)
  );
```

## P22 (R15-Q8) — Exit-family naming rename (hierarchical)

**Resolution:** **Option C hierarchical** (matches lean).

**Renames** (3 values, per-limit taxonomy):

| Old | New | Used since |
|---|---|---|
| `evaluated:reducing_exit` | `evaluated:exit:reducing` | v1.6 |
| `evaluated:reducing_exit_complete` | `evaluated:exit:complete` | v1.10 |
| `evaluated:exit_with_flip` | `evaluated:exit:flip` | v1.11 |

**Rationale:** Matches the colon-as-separator hierarchy already used for `cb_*`
values (`cb_throttle:applied`, `cb_block_new_risk:risk_reducer_exempted`,
etc.). Hierarchy depth: `<surface>:<family>:<variant>`. The `evaluated:exit:*`
group is now a clean family.

**Scope:** This rename applies ONLY to the exit family. Other `evaluated:*`
values (`within_limits`, `limit_breached`, `cancel_no_effect`) are
single-modifier and not renamed. Extending the hierarchy to all values would
be scope creep beyond R15-Q8's prescription.

**Audit symmetry:**
- P10 verdict_raw: doesn't reference exit values directly. ✓
- Cancel outcome matrix: updated below (Final outcome matrix).
- `is_genuinely_risk_reducing` predicate: doesn't reference these reasons. ✓
- evaluation_inputs: typed FK lineage doesn't reference reason values. ✓

**SQL-phase impact:** CHECK constraint on
`evaluation_limit_results.result_reason` enumerates all permitted values
including the renamed three.

## P23 (R15-Q9) — Bucket D `unknown` strictness = D1

**Resolution:** **D1 — degraded path, same handling as
`stale_needs_reconciliation`** (matches lean).

**Confirmed semantics for `unknown`:**
- Treated identically to `stale_needs_reconciliation` in P14 / P9 logic.
- `target_remaining_qty := NULL`
- `is_genuinely_risk_reducing := false`
- `result_reason := 'insufficient_inputs:target_state_indeterminate'`
- `blocking := false`
- Routes to `verdict_raw='degraded'` →
  - LIVE: `verdict_effective='blocked'` (fail-closed)
  - non-LIVE: `verdict_effective='allowed'` (fail-open)

**Rejected alternatives:**
- D2 (RAISE on `unknown` only): operationally inconsistent with the rest of
  the input-degradation model. If `unknown` is a data-integrity bug, the
  audit trail (degraded row + insufficient_inputs reason) is the right place
  to surface it; halting the evaluator is over-strong.
- D3 (treat as terminal/no-effect): unsafe. Could allow cancels-with-effect
  against actually-live orders to be classified as no-op.

**Operator follow-up note (not part of structural design):** A monitoring
query on `evaluation_limit_results` filtering for
`result_reason='insufficient_inputs:target_state_indeterminate'` will surface
any production occurrence of `unknown` state and flag for investigation.
Worth adding to the watchdog after SQL deployment.

## Final taxonomies (consolidated, canonical for SQL)

### Per-limit: `evaluation_limit_results.result_reason` (NOT NULL)

| Value | Meaning | Severity bucket |
|---|---|---|
| `evaluated:within_limits` | Limit not breached. | `within_limits` |
| `evaluated:limit_breached` | Limit breached, predicate=false → blocking. | `breach` or `critical` |
| `evaluated:exit:reducing` | Limit breached, predicate=true → escape hatch, non-blocking. | `breach` or `critical` |
| `evaluated:exit:complete` | Action exited fully (Q'=0), predicate=true. | `breach` or `critical` |
| `evaluated:exit:flip` | Sign-flip lands breached on opposite side. | `breach` or `critical` |
| `evaluated:cancel_no_effect` | Cancel target has zero remaining (zero-rem or terminal-state). | `within_limits` |
| `insufficient_inputs:missing` | Required input missing. | `insufficient_inputs` |
| `insufficient_inputs:stale` | Required input present but exceeds staleness bound. | `insufficient_inputs` |
| `insufficient_inputs:target_state_indeterminate` | Cancel target in Bucket D state (`stale_needs_reconciliation` or `unknown`). | `insufficient_inputs` |

**9 values.** SQL CHECK constraint enumerates these exhaustively.

**Invariants:**
- `severity_bucket = 'insufficient_inputs'` ⟺ `result_reason LIKE 'insufficient_inputs:%'`
- `severity_bucket = 'insufficient_inputs'` ⟹ `blocking = false`
- `blocking = true` ⟹ `severity_bucket IN ('breach', 'critical')`

### Per-evaluation CB: `evaluations.circuit_breaker_result_reason` (NULL allowed)

| Value | Meaning | Verdict contribution |
|---|---|---|
| (NULL) | No CB active. | none |
| `cb_warn_only:applied` | warn_only CB active. | none (allowed) |
| `cb_throttle:applied` | Throttle CB active and applied. | none (allowed; runtime applies rate-limit) |
| `cb_throttle:cancel_exempted` | Throttle active, source_type=cancel exempted. | none (allowed) |
| `cb_throttle:risk_reducer_exempted` | Throttle active, predicate=true exempted. | none (allowed) |
| `cb_block_new_risk:applied` | block_new_risk active, not exempted. | blocked |
| `cb_block_new_risk:risk_reducer_exempted` | block_new_risk active, predicate=true exempted. | none (allowed) |
| `cb_hard_stop:applied` | hard_stop active. | blocked (dominates) |
| `insufficient_inputs:cb_missing` | CB resolution failed due to missing inputs. | degraded |

**9 values incl. NULL.** SQL CHECK constraint:
```sql
CHECK (circuit_breaker_result_reason IS NULL
       OR circuit_breaker_result_reason IN (
         'cb_warn_only:applied',
         'cb_throttle:applied',
         'cb_throttle:cancel_exempted',
         'cb_throttle:risk_reducer_exempted',
         'cb_block_new_risk:applied',
         'cb_block_new_risk:risk_reducer_exempted',
         'cb_hard_stop:applied',
         'insufficient_inputs:cb_missing'
       ))
```

## Final cancel-flow outcome matrix (v1.15, consolidated)

For `source_type='cancel'` against an instrument-scope max_position_quantity
limit, in `risk_environment='LIVE'`:

| Target state | Bucket | target_remaining_qty | predicate | limit row reason | limit row blocking | severity_bucket | verdict_raw | verdict_effective |
|---|---|---|---|---|---|---|---|---|
| working/submitted/etc., target reducing, breaching | A | > 0 | false | `evaluated:limit_breached` | true | breach | blocked | blocked |
| working/submitted/etc., target adding, breaching | A | > 0 | true | `evaluated:exit:reducing` | false | breach | allowed | allowed |
| working, target reducing toward Q'=0 | A | > 0 | true | `evaluated:exit:complete` | false | breach | allowed | allowed |
| working, sign-flip into breach | A | > 0 | false | `evaluated:exit:flip` | true | breach | blocked | blocked |
| working, within limit | A | > 0 | (any) | `evaluated:within_limits` | false | within_limits | allowed | allowed |
| canceled/filled/rejected/expired/failed_submit | B | 0 | false | `evaluated:cancel_no_effect` | false | within_limits | allowed | allowed |
| Bucket B + cb_hard_stop active | B | 0 | false | `evaluated:cancel_no_effect` | false | within_limits | blocked (via CB) | blocked |
| stale_needs_reconciliation / unknown | D | NULL | false | `insufficient_inputs:target_state_indeterminate` | false | insufficient_inputs | degraded | blocked |

For `risk_environment != 'LIVE'`, the only difference is the last row:
`degraded → allowed`. All other rows are environment-invariant.

**Test oracle for SQL phase.** SQL test cases should cover each row × {LIVE,
SHADOW, REPLAY, BACKTEST} with at least one fixture per row. Estimated 32+
test fixtures for the cancel branch alone.

## Process notes (for future migrations)

R6-R15 produced 24 patches total (avg 2.4/round). Patterns worth carrying
forward:

1. **Holistic audit principle** (R10): when applying a fix, audit symmetric
   applications across all enforcement surfaces. Final audit list:
   drawdown / instrument-limits / CB block_new_risk / CB throttle / per-limit
   reason / CB reason / verdict_raw severity_bucket guard.
2. **Explicit supersession** (R13 self-discovered): each new vN explicitly
   marks deprecated sections of vN-1, not merely acknowledged.
3. **Cross-version state checking** (R14 self-discovered): when reviewer
   feedback claims a state, cross-check against prior-round resolutions
   before patching. Avoids re-litigating resolved items.
4. **Verify-then-draft** workflow: when a patch depends on schema state, run
   verification queries first, then draft against confirmed branches with
   alternates flagged. Saved 2+ rounds during R11-R13.
